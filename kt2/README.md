## Практика 1 — Базовая классификация

**Файл:** `1/lab1.ipynb`  
**Задача:** Предсказать, уйдёт ли клиент телеком-компании (отток/churn).  
**Датасет:** `telecom_churn.csv` — таблица с 3333 клиентами и 21 признаком (тарифы, звонки, платежи и т.д.)  
**Модели:** Логистическая регрессия, Дерево решений, KNN

### Что делает код (пошагово)

#### 1. Загрузка и разведочный анализ (EDA)
```python
df = pd.read_csv('telecom_churn.csv')
```
Загружаем CSV-файл с данными. Затем проводим разведочный анализ:
- **Размер датасета** — 3333 строки (клиента) × 21 столбец (признак).
- **Типы данных** — 16 числовых столбцов (доход, минуты разговоров, количество звонков) и 5 категориальных (штат, номер телефона, тариф "да/нет", отток).
- **Пропуски** — проверяем пустые ячейки. В данном датасете пропусков нет (все 0), поэтому не требуется заполнение или удаление.
- **Распределение целевой переменной** — сколько клиентов ушло / осталось: **2850 лояльных** (False) vs **483 ушедших** (True). Это значит, что данные **несбалансированы** — ушедших почти в 6 раз меньше.

> **Зачем нужен EDA?** Прежде чем строить модель, нужно понять данные. Если пропусков много — нужно их заполнить. Если классы сильно несбалансированы — модель может «хитрить» и всегда предсказывать класс большинства, получая высокую точность, но не находя реальных уходящих клиентов.

#### 2. Подготовка данных (предобработка)
```python
X = df.drop('churn', axis=1)          # убираем целевую переменную из признаков
y = df['churn'].astype(int)           # целевая переменная: 0 = остался, 1 = ушёл

preprocessor = ColumnTransformer(transformers=[
    ('num', StandardScaler(), numeric_features),
    ('cat', OneHotEncoder(handle_unknown='ignore', drop='first', sparse_output=False), categorical_features)
])
```
- **StandardScaler** — приводит числовые признаки к одному масштабу (среднее = 0, стандартное отклонение = 1). Без этого модели вроде KNN будут считать, что «доход в тысячах» важнее «количества звонков в единицах» просто из-за разницы в масштабе чисел.
- **OneHotEncoder** — превращает категориальные признаки (штат, тариф «да/нет») в числа. Например, столбец «международный план» с двумя значениями превращается в бинарный столбец (1 или 0). Параметр `drop='first'` убирает один из столбцов, т.к. он избыточен (если «да» = 0, значит автоматически «нет» = 1).
- **handle_unknown='ignore'** — если в тестовых данных встретится категория, которой не было в обучающих, модель просто проигнорирует её, а не упадёт с ошибкой.

> **Зачем ColumnTransformer?** Он позволяет применить разную обработку к разным типам столбцов в одном шаге — числовые масштабируются, категориальные кодируются.

#### 3. Разделение на обучающую и тестовую выборки
```python
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, random_state=42)
```
- **75% данных** (2499 строк) уходит на обучение модели.
- **25% данных** (834 строки) — на проверку (тест). Мы «прячем» эти данные от модели, чтобы потом честно оценить, насколько хорошо она предсказывает на **новых** данных, которых никогда не видела.
- `random_state=42` — фиксируем генератор случайных чисел, чтобы при повторном запуске разбиение было одинаковым и результаты воспроизводились.

#### 4. Обучение трёх моделей
```python
models = {
    'Логистическая регрессия': LogisticRegression(random_state=42, max_iter=1000),
    'Дерево решений': DecisionTreeClassifier(random_state=42),
    'KNN': KNeighborsClassifier()
}
```

| Модель | Суть простыми словами | Плюсы | Минусы |
|---|---|---|---|
| **Логистическая регрессия** | Проводит разделяющую «линию» (гиперплоскость) между классами. Несмотря на название — это модель **классификации**, а не регрессии. | Быстрая, интерпретируемая, хорошо работает при линейных зависимостях | Не ловит сложные нелинейные закономерности |
| **Дерево решений** | Строит дерево вопросов «если-то»: «Звонков в сервис > 3?» → «Да» → «Минут > 200?» → ... Идёт по веткам до конечного листа с ответом. | Интуитивно понятна, не требует масштабирования | Склонна к переобучению |
| **KNN (K ближайших соседей)** | Для нового клиента находит K самых похожих клиентов из обучающей выборки и «голосует» — к какому классу отнести нового. | Простая идея, не нужно «обучать» в традиционном смысле | Медленная на больших данных, чувствительна к масштабу |

Каждая модель **оборачивается в Pipeline** — цепочку «предобработка → модель», чтобы при вызове `.fit()` и `.predict()` всё происходило автоматически в правильном порядке.

#### 5. Оценка: Accuracy и Confusion Matrix

**Результаты:**
| Модель | Accuracy |
|---|---|
| Логистическая регрессия | 0.8597 |
| **Дерево решений** | **0.9508** |
| KNN | 0.8837 |

- **Accuracy (точность)** — доля правильных ответов из всех предсказаний. Лучшая модель — **дерево решений** с 95%.
- **Столбчатая диаграмма** (`accuracy_comparison.png`) — наглядно сравниваем точность.

**Confusion Matrix** (матрица ошибок) для лучшей модели (дерева решений):
```
[[698  11]
 [ 30  95]]
```
- **698 True Negatives** — правильно определили лояльных.
- **95 True Positives** — правильно определили ушедших.
- **11 False Positives** — ошибочно предсказали уход (клиент остался) — бизнесу это менее опасно.
- **30 False Negatives** — **самая опасная ошибка** — не заметили, что клиент уйдёт. Компания теряет деньги.

> **Почему дальше мы улучшаем модель?** Хотя Accuracy = 95%, модель пропускает 30 из 125 уходящих клиентов (Recall ≈ 76%). Именно поэтому в Практике 2 мы будем бороться за повышение **Recall** (полноты) — способности модели **находить** ушедших.

## Практика 2 — Настройка и улучшение модели

**Файл:** `2/lab2.ipynb`  
**Задача:** Улучшить модель из Практики 1 — сделать так, чтобы она **лучше находила уходящих клиентов**, даже если при этом немного просядет общая точность.  
**Датасет:** тот же `telecom_churn.csv`  
**Методы:** SMOTE, GridSearchCV, ROC-кривая, Feature Importance

### Проблема, которую мы решаем

В Практике 1 модель хорошо предсказывала лояльных (класс 0), но **плохо находила ушедших** (класс 1). Это классическая проблема **дисбаланса классов** — ушедших клиентов в датасете примерно в 6 раз меньше, чем лояльных. Модель «ленится» и чаще предсказывает класс большинства, получая высокую Accuracy, но низкий Recall на меньшинстве.

### Что делает код (пошагово)

#### 1. Загрузка данных и предобработка
Данные загружаются и подготавливаются **точно так же**, как в Практике 1 — тот же `ColumnTransformer` с `StandardScaler` и `OneHotEncoder`. Разделение: 75% train / 25% test с `random_state=42`.

#### 2. Диагностика старой модели
```python
old_model = ImbPipeline(steps=[
    ('preprocessor', preprocessor),
    ('classifier', DecisionTreeClassifier(random_state=42))
])
```
Обучаем **ту же модель** (дерево решений без настроек), но теперь используем `ImbPipeline` из библиотеки `imbalanced-learn` (он нужен для совместимости с SMOTE). Печатаем полный `classification_report`:
- **Precision** — «из всех, кого модель назвала ушедшими, сколько реально ушли?» Если = 0.9, значит 90% предупреждений об оттоке были верными.
- **Recall** — «из всех реально ушедших, скольких модель нашла?» Если = 0.76, значит 24% уходящих клиентов мы **пропустили**.
- **F1-Score** — среднее гармоническое Precision и Recall — одно число, которое балансирует оба показателя. Используется для оптимизации.

Запоминаем все метрики старой модели для финального сравнения.

#### 3. SMOTE + GridSearchCV — два ключевых приёма

**SMOTE (Synthetic Minority Over-sampling Technique)**
```python
('smote', SMOTE(random_state=42))
```
- **Проблема:** ушедших клиентов мало → модель их плохо учит.
- **Решение:** SMOTE **генерирует синтетические** (искусственные) примеры ушедших клиентов, чтобы выровнять количество обоих классов.
- **Как это работает:** SMOTE берёт реального ушедшего клиента, находит его ближайших «соседей» среди других ушедших, и создаёт **новую искусственную точку** на линии между ними в пространстве признаков. Это не тупое копирование, а генерация похожих, но новых данных.

> **Почему не просто копируем?** Простое дублирование (oversampling) создаёт точные копии, и модель может их просто «заучить». SMOTE создаёт новые, но **похожие** примеры — это лучше для обобщающей способности модели.

> **Важно:** SMOTE применяется **только к обучающей выборке**. Тестовая выборка остаётся «честной» — без синтетических данных.

**GridSearchCV (Поиск по сетке гиперпараметров)**
```python
param_grid = {
    'classifier__max_depth': [5, 10, 15, 20, None],
    'classifier__min_samples_split': [2, 10, 20],
    'classifier__min_samples_leaf': [1, 5, 10],
    'classifier__criterion': ['gini', 'entropy']
}
```
У дерева решений есть **гиперпараметры** — настройки, которые мы задаём **до** обучения:
- `max_depth` — максимальная глубина дерева. Чем глубже — тем сложнее модель, но тем выше риск переобучения. `None` = неограниченная глубина.
- `min_samples_split` — минимальное число примеров, чтобы узел дерева поделился дальше. Чем больше — тем «осторожнее» дерево.
- `min_samples_leaf` — минимальное число примеров в конечном листе. Ограничивает слишком мелкие группы.
- `criterion` — критерий оценки качества разделения: `gini` (индекс Джини) или `entropy` (энтропия). Оба измеряют «чистоту» получающихся групп.

GridSearchCV **перебирает все комбинации** этих параметров (5 × 3 × 3 × 2 = **90 вариантов**) и для каждого проводит **5-Fold кросс-валидацию**:
1. Обучающая выборка делится на 5 частей.
2. Модель учится на 4 частях, проверяется на 5-й.
3. Это повторяется 5 раз (каждая часть побывала тестовой).
4. Итого: 90 × 5 = **450 обучений**.

Цель — найти комбинацию с лучшим **F1-Score** (параметр `scoring='f1'`).

Параметр `n_jobs=-1` означает использование всех доступных ядер процессора для ускорения.

#### 4. Финальная оценка улучшенной модели
Берём лучшую модель (`grid_search.best_estimator_`) и оцениваем на тестовой выборке:
- Вычисляем Accuracy, Precision, Recall, F1 для новой модели.
- Строим **Confusion Matrix** (`confusion_matrix_optimized.png`) — сравниваем с матрицей старой модели.

**ROC-кривая и AUC** (`roc_curve.png`):
```python
fpr, tpr, thresholds = roc_curve(y_test, y_prob_new)
roc_auc = auc(fpr, tpr)
```
- ROC-кривая показывает, как модель балансирует между **True Positive Rate** (Recall) и **False Positive Rate** при разных порогах срабатывания.
- **AUC (Area Under Curve)** — площадь под ROC-кривой: 0.5 = случайное угадывание, 1.0 = идеальное разделение.

**Feature Importance** (`feature_importance.png`):
- Для дерева решений автоматически рассчитывается важность каждого признака. Выводим топ-10 самых влиятельных.
- Это помогает бизнесу понять: **какие факторы больше всего влияют на отток?** (например, количество звонков в сервисную службу).

#### 5. Таблица сравнения
Финальная таблица сравнивает старую и новую модель по всем метрикам. **Цель** — Recall (способность находить уходящих) должен вырасти при сохранении приемлемого общего качества.

## Практика 3 — Мультиклассовая классификация

**Файл:** `3/lab3.ipynb`  
**Задача:** Перейти от бинарной классификации (2 класса: ушёл/не ушёл) к **мультиклассовой** (3 и более классов) — предсказывать сорт вина по химическим признакам.  
**Датасет:** `Wine` — встроенный датасет scikit-learn  
**Модели:** RandomForest, LogisticRegression, RandomForest (Balanced)

### Чем отличается от предыдущих работ?

В Практиках 1–2 мы предсказывали **два класса** (ушёл или нет). Здесь — **три класса** (три сорта вина). Это требует:
- Метрик, которые корректно работают с несколькими классами (**Macro F1** вместо обычного F1).
- Стратегий для балансировки более чем двух классов.
- Другого подхода к построению матрицы ошибок (3×3 вместо 2×2).

### Что делает код (пошагово)

#### 1. Датасет Wine
```python
wine = datasets.load_wine()
```
- **178 образцов** вин, **13 химических признаков** (алкоголь, яблочная кислота, зольность, магний, флавоноиды, пролин и т.д.).
- **3 класса:** `class_0` (59 шт.), `class_1` (71 шт.), `class_2` (48 шт.) — три разных сорта итальянского вина.
- Классы **не идеально сбалансированы** (59 vs 71 vs 48), но дисбаланс не критичный.
- Датасет относительно лёгкий, но хорошо подходит для изучения мультиклассовых подходов.

#### 2. Разведочный анализ (EDA)
- Считаем количество образцов в каждом классе → строим **столбчатую диаграмму** (`class_distribution.png`).
- Визуально убеждаемся, что дисбаланс присутствует (class_2 — самый маленький), но не экстремальный.

#### 3. Разделение данных
```python
X_train, X_test, y_train, y_test = train_test_split(
    X, y, test_size=0.3, random_state=42, stratify=y
)
```
- `stratify=y` — **ключевой параметр**: гарантирует, что в обучающей и тестовой выборках **процент каждого класса одинаковый**. Без этого в маленьком тестовом наборе один из классов может оказаться недопредставленным.
- 70% train / 30% test.

#### 4. Обучение двух базовых моделей

| Модель | Суть |
|---|---|
| **RandomForest (Base)** | Ансамбль из множества деревьев решений. Каждое дерево обучается на случайной подвыборке данных и признаков. Итоговый ответ — голосованием большинства. Более устойчивая и точная, чем одно дерево. |
| **LogisticRegression** | Линейная модель, которая для мультикласса использует стратегию «один против всех» (one-vs-rest): строит отдельную модель для каждого класса. `max_iter=10000` увеличивает максимальное число итераций для гарантии сходимости. |

Для каждой модели строится **Confusion Matrix** и выводится полный **classification_report**.

#### 5. Балансировка весов (class_weight='balanced')
```python
rf_balanced = RandomForestClassifier(random_state=42, class_weight='balanced')
```
Параметр `class_weight='balanced'` автоматически увеличивает «штраф» за ошибки на редких классах. Если class_2 встречается в 1.5 раза реже class_1, то модель получает в 1.5 раза больший «штраф» за ошибку на class_2. Модель начинает уделять больше внимания малочисленному классу.

> **Отличие от SMOTE:** SMOTE **создаёт новые данные** (увеличивает выборку), а `class_weight` **меняет функцию потерь** (изменяет штрафы). Оба метода борются с дисбалансом, но разными способами. `class_weight` проще и не требует отдельной библиотеки.

#### 6. Метрика — Macro F1-Score
```python
f1_score(y_test, y_pred, average='macro')
```
- **Macro F1** — считает F1 для **каждого класса отдельно**, потом берёт **простое среднее** (не взвешенное по количеству).
- Это **справедливая** метрика для мультикласса: она не позволяет модели «прятаться» за большой класс. Каждый класс одинаково важен.

#### 7. Итоговое сравнение
Все три модели сравниваются по Macro-F1 на тестовой выборке. Выводится анализ:
- Какой класс «труднее» для модели.
- Как `class_weight='balanced'` повлиял на результат.
- Рекомендация для финального прототипа.

## Практика 4 — Оптимизация мультиклассовой модели

**Файл:** `4/lab4.ipynb`  
**Задача:** Улучшить модель из Практики 3 с помощью продвинутых методов — **SMOTE, кривых обучения, Pipeline + GridSearchCV**.  
**Датасет:** тот же `Wine`  
**Методы:** SMOTE, Learning Curve, Pipeline + GridSearchCV

### Связь с Практикой 3

Практика 3 дала нам базовые модели. Теперь мы применяем **те же техники**, что использовали в Практике 2 (SMOTE, GridSearchCV), но уже к **мультиклассовой** задаче. Плюс добавляем **кривые обучения** для диагностики.

### Что делает код (пошагово)

#### 1. SMOTE для мультикласса
```python
smote = SMOTE(random_state=42)
X_train_smote, y_train_smote = smote.fit_resample(X_train, y_train)
```
- SMOTE работает и для 3+ классов: он доводит **каждый** класс до размера самого большого.
- Печатаем распределение **до** и **после**. До: [41, 50, 33] → после: все классы имеют по 50 примеров.
- Модель RandomForest обучается на сбалансированных данных.

#### 2. Кривые обучения (Learning Curve) — диагностика модели
```python
train_sizes, train_scores, test_scores = learning_curve(
    RandomForestClassifier(random_state=42), X_train_smote, y_train_smote,
    cv=5, scoring='f1_macro', n_jobs=-1, train_sizes=np.linspace(0.1, 1.0, 5)
)
```
Строим график (`learning_curve.png`): как меняется качество модели при увеличении объёма обучающих данных:
- **Обучающая кривая** (синяя) — качество на тех данных, на которых учились.
- **Валидационная кривая** (оранжевая, пунктир) — качество на данных, которые модель не видела.
- Заливка вокруг линий — стандартное отклонение (разброс между складками кросс-валидации).

**Как читать график (важно для защиты!):**
| Ситуация на графике | Диагноз | Что делать |
|---|---|---|
| Обе кривые высоко и рядом | Модель хорошая, данных достаточно | Всё ок |
| Обучающая высоко, валидационная низко | **Переобучение** — модель заучила данные | Упростить модель, добавить регуляризацию, собрать больше данных |
| Обе кривые низко | **Недообучение** — модель слишком простая | Усложнить модель, добавить признаки |

В нашем случае: обучающая ≈ 1.0, валидационная растёт — **лёгкое переобучение**, но в целом модель адекватная.

#### 3. Pipeline + GridSearchCV
```python
pipeline = ImbPipeline(steps=[
    ('scaler', StandardScaler()),
    ('rf', RandomForestClassifier(random_state=42))
])
param_grid = {
    'rf__max_depth': [5, 10, None],
    'rf__n_estimators': [50, 100]
}
cv = StratifiedKFold(n_splits=3, shuffle=True, random_state=42)
grid_search = GridSearchCV(pipeline, param_grid, cv=cv, scoring='f1_macro', n_jobs=-1, verbose=1)
```
- **StandardScaler** — нормализация всех признаков (приведение к среднему 0; разбросу 1). Важно для стабильности моделей.
- **n_estimators** — количество деревьев в случайном лесу. Больше = точнее, но медленнее.
- **StratifiedKFold** — кросс-валидация, которая **сохраняет пропорции классов** в каждой «складке». Для мультикласса это критично.
- Перебираем 3 × 2 = 6 комбинаций, каждую проверяем 3-Fold кросс-валидацией = **18 обучений**.

#### 4. Итоговая таблица
Сравниваем Macro-F1 на тестовой выборке по трём этапам:
1. **Базовая модель** (из практики 3) — ориентир.
2. **+ SMOTE** (без тюнинга) — как балансировка данных влияет.
3. **+ Pipeline + GridSearchCV** — комплексная оптимизация.

## Практика 5 — Регрессия

**Файл:** `5/lab5.ipynb`  
**Задача:** Перейти от **классификации** (предсказание категории) к **регрессии** (предсказание числа) — прогнозировать стоимость дома.  
**Датасет:** `California Housing` — встроенный датасет scikit-learn с данными о 20 640 объектах недвижимости в Калифорнии.  
**Модели:** LinearRegression, DecisionTreeRegressor

### Чем регрессия отличается от классификации?

| | Классификация | Регрессия |
|---|---|---|
| **Что предсказываем** | Категорию (ушёл/не ушёл, сорт вина) | Числовое значение (цена дома в сотнях тысяч $) |
| **Метрики** | Accuracy, F1, Recall | MAE, MSE, R² |
| **Пример ответа** | «Этот клиент уйдёт» | «Этот дом стоит $250k» |

### Что делает код (пошагово)

#### 1. Исследование целевой переменной
```python
sns.histplot(y, bins=50, kde=True, color='teal')
```
Строим **гистограмму распределения цен** (`target_histogram.png`):
- Ось X — стоимость в единицах $100k (т.е. значение 2.0 = $200k).
- Видим, что большинство домов стоят от $100k до $300k, но есть «хвост» дорогих домов.
- Кривая KDE (ядерное сглаживание плотности) показывает общую форму распределения.

#### 2. Выбор признаков и визуализация связей
```python
selected_features = ['MedInc', 'AveRooms']
```
Берём **только два признака** для наглядности:
- **MedInc** — медианный доход в районе (самый сильный предиктор цены).
- **AveRooms** — среднее количество комнат.

Строим **scatter-plot** (`features_vs_target.png`) — как каждый признак связан с ценой. Видим:
- Доход явно коррелирует с ценой (чем выше доход → тем дороже).
- Среднее число комнат — связь менее очевидна и зашумлена.

#### 3. Линейная регрессия и интерпретация
```python
lr = LinearRegression()
lr.fit(X_train, y_train)
```
Самая простая модель регрессии: ищет **прямую линию** (в многомерном пространстве — гиперплоскость), которая лучше всего приближает данные. Формула: `Цена = a₁×Доход + a₂×Комнаты + b`.

**Метрики регрессии (важно для защиты!):**
| Метрика | Что означает | Как интерпретировать |
|---|---|---|
| **MAE** (Mean Absolute Error) | Средняя абсолютная ошибка | «В среднем мы ошибаемся на X единиц» |
| **MSE** (Mean Squared Error) | Средняя квадратичная ошибка | Штрафует большие ошибки сильнее, чем MAE |
| **R²** (коэффициент детерминации) | Доля дисперсии данных, объяснённая моделью | 1.0 = идеально; 0 = модель не лучше простого среднего; <0 = хуже среднего |

**Интерпретация коэффициентов (бизнес-значение):**
- Печатаем, на сколько изменится предсказанная цена при изменении каждого признака на 1 единицу.
- Например: увеличение `AveRooms` на 1 → цена меняется на `coef × $100k`. Это позволяет давать рекомендации заказчику.

#### 4. Сравнение с деревом решений
```python
dt = DecisionTreeRegressor(max_depth=3, random_state=42)
```
- Дерево решений работает и для регрессии: вместо «голосования» по классам оно **усредняет значения** в каждом листе.
- `max_depth=3` — ограничиваем глубину до 3-х уровней, чтобы дерево не переобучалось.

#### 5. Итоговая таблица и scatter-plot
- Таблица сравнивает MAE и R² обеих моделей.
- **Scatter-plot** (`real_vs_pred_scatter.png`) — реальная цена vs предсказанная.
  - Идеальная модель: все точки на диагонали `y = x`.
  - Чем дальше точки от диагонали — тем больше ошибка.

## Практика 6 — Борьба с переобучением

**Файл:** `6/lab6.ipynb`  
**Задача:** Продемонстрировать переобучение на конкретном примере и показать, как **Ridge-регуляризация** помогает с ним бороться.  
**Датасет:** тот же `California Housing`  
**Методы:** Добавление шумовых признаков, Ridge-регрессия

### Что такое переобучение (overfitting)?

Модель **слишком хорошо выучила обучающие данные** — включая шум и случайные паттерны — но плохо работает на новых данных. Аналогия: студент, который заучил ответы прошлогодних билетов, но не понимает предмет и проваливается на новых вопросах.

**Признак переобучения:** R² на обучающей выборке **высокий**, а R² на тестовой — **значительно ниже**.

### Что делает код (пошагово)

#### 1. Базовая модель (ориентир)
```python
scaler = StandardScaler()
X_train_scaled = scaler.fit_transform(X_train)
X_test_scaled = scaler.transform(X_test)

lr_base = LinearRegression()
lr_base.fit(X_train_scaled, y_train)
r2_base_test = r2_score(y_test, lr_base.predict(X_test_scaled))
```
- Нормализуем данные с помощью `StandardScaler` (обучаем на train, применяем к test — **не наоборот**, чтобы не «подсматривать» в будущее).
- Обучаем на **всех 8 реальных признаках**, запоминаем R² на тесте ≈ **0.5958** как **базовый уровень**.

#### 2. Искусственная «поломка» — добавляем шум
```python
np.random.seed(42)
noise_train = np.random.normal(0, 1, (X_train.shape[0], 30))
X_train_noisy = np.hstack((X_train, noise_train))
```
- Генерируем **30 полностью случайных столбцов** (нормальное распределение) и добавляем к данным.
- Теперь вместо 8 признаков → **38**, но 30 из них — **чистый шум**, не связанный с ценой дома.

> **Зачем?** Это моделирует реальную ситуацию, когда в данных есть признаки, которые **никак не связаны** с целевой переменной (например, ID клиента, случайные коды). Линейная регрессия всё равно попытается найти в них «закономерности».

#### 3. Демонстрация переобучения
```python
lr_broken = LinearRegression()
lr_broken.fit(X_train_noisy_scaled, y_train)
```
- R² на **обучающей** выборке: ≈ 0.6099 — модель «подстроилась» под шум в тренировочных данных.
- R² на **тестовой** выборке: ≈ 0.5950 — **падение**! Случайные коэффициенты, подогнанные под шум в train, не работают на test, потому что шум в test — **другой**.

**Ключевое наблюдение:** R² на train **выше** R² на test — классический признак переобучения.

#### 4. Спасение — Ridge-регрессия
```python
ridge = Ridge(alpha=a, random_state=42)
```

**Ridge-регрессия** — это линейная регрессия с **L2-регуляризацией**:
- Обычная линейная регрессия минимизирует **только ошибку**: `Loss = MSE`.
- Ridge минимизирует: `Loss = MSE + α × Σ(коэффициентов²)`.
- Добавка `α × Σ(коэффициентов²)` **штрафует большие коэффициенты**. Модель вынуждена «экономить» — назначает большие коэффициенты только тем признакам, которые действительно полезны.

Параметр **α (alpha)** — «сила» регуляризации:
| Значение α | Эффект |
|---|---|
| α = 0 | Обычная линейная регрессия (без штрафа) |
| α маленькое (0.1) | Слабая регуляризация — почти как без неё |
| α среднее (10–100) | Баланс между точностью и простотой |
| α большое (1000+) | Очень сильная регуляризация — все коэффициенты почти нулевые, модель слишком простая |

**Как это помогает с шумом?** Шумовые признаки — случайные, они не помогают предсказывать цену. Ridge «душит» коэффициенты для бесполезных признаков → модель их фактически **игнорирует** → переобучение уменьшается.

#### 5. Перебор alpha и поиск оптимального
```python
alphas = [0.1, 1, 10, 100, 1000]
for a in alphas:
    ridge = Ridge(alpha=a, random_state=42)
    ridge.fit(X_train_noisy_scaled, y_train)
    r2_test_ridge = r2_score(y_test, ridge.predict(X_test_noisy_scaled))
```
Перебираем 5 значений alpha и для каждого записываем R² на тесте. Видим, что **alpha=100** даёт лучший результат.

#### 6. Финальная таблица
| Модель/условия | R² на test | Комментарий |
|---|---|---|
| Базовая (без шума) | ~0.5958 | Наш ориентир |
| Сломанная (LinearReg + шум) | ~0.5950 | Переобучение! |
| Исправленная (Ridge, alpha=100) | ~0.5960 | Спасение! Качество восстановлено |

> **Вывод:** Ridge-регрессия **полностью компенсировала** влияние 30 шумовых признаков и вернула качество к базовому уровню.

## Общая логика всех работ

```
Практика 1: Знакомство → выбираем лучшую из трёх простых моделей (бинарная классификация)
       ↓
Практика 2: Улучшаем → SMOTE (борьба с дисбалансом) + GridSearch (подбор параметров)
       ↓
Практика 3: Усложняем задачу → 3 класса вместо 2-х (мультиклассовая классификация)
       ↓
Практика 4: Оптимизируем мультикласс → кривые обучения (диагностика) + комплексный тюнинг
       ↓
Практика 5: Меняем тип задачи → регрессия (предсказываем число, а не категорию)
       ↓
Практика 6: Боремся с переобучением → Ridge-регуляризация
```

**Красная нить:** на протяжении всех практик мы двигаемся от простого к сложному:
- Простая модель → настройка → оптимизация.
- Бинарная классификация → мультикласс → регрессия.
- Базовые метрики (Accuracy) → продвинутые (F1, ROC-AUC, R²).
- Без борьбы с проблемами → SMOTE (дисбаланс) → Ridge (переобучение).

## Словарь терминов

| Термин | Простое объяснение |
|---|---|
| **Accuracy** | Доля правильных ответов из всех |
| **Precision** | «Из тех, кого назвали ушедшими, сколько реально ушли?» |
| **Recall** | «Из всех реально ушедших, скольких нашли?» |
| **F1-Score** | Баланс между Precision и Recall (среднее гармоническое) |
| **Macro F1** | Средний F1 по всем классам (каждый класс одинаково важен) |
| **Confusion Matrix** | Таблица: что модель предсказала vs что было на самом деле |
| **ROC-AUC** | Площадь под кривой разделения классов (0.5 = случайно, 1.0 = идеально) |
| **MAE** | Средняя абсолютная ошибка в регрессии |
| **MSE** | Средняя квадратичная ошибка (штрафует большие ошибки сильнее) |
| **R²** | Насколько модель объясняет данные (0 = плохо, 1 = отлично) |
| **SMOTE** | Генерация синтетических примеров для балансировки классов |
| **GridSearchCV** | Автоматический перебор параметров модели с кросс-валидацией |
| **Кросс-валидация** | Проверка модели на нескольких разбиениях данных (fold'ах) |
| **Pipeline** | Цепочка шагов «предобработка → модель» в одном объекте |
| **Переобучение** | Модель запомнила данные вместо того, чтобы научиться обобщать |
| **Регуляризация** | Штраф за сложность модели, помогает против переобучения |
| **Ridge** | L2-регуляризация — штрафует сумму квадратов коэффициентов |
| **StandardScaler** | Нормализация данных к среднему 0 и стандартному отклонению 1 |
| **OneHotEncoder** | Превращение категорий (текст) в числовые столбцы (0/1) |
| **Feature Importance** | Рейтинг признаков по степени влияния на предсказание |
| **Learning Curve** | График зависимости качества модели от объёма данных |
| **Дисбаланс классов** | Когда одного класса сильно больше другого → модель «ленится» |
