## Практика 1 — Базовая классификация

**Файл:** `1/main.py`  
**Задача:** Предсказать, уйдёт ли клиент телеком-компании (отток/churn).  
**Датасет:** `telecom_churn.csv` — таблица с данными о клиентах (тарифы, звонки, платежи и т.д.)

### Что делает код (пошагово)

#### 1. Загрузка и разведочный анализ
```python
df = pd.read_csv('telecom_churn.csv')
```
Загружаем CSV-файл с данными. Затем смотрим:
- **Размер** — сколько строк (клиентов) и столбцов (признаков).
- **Типы данных** — какие столбцы числовые (доход, минуты), какие категориальные (штат, тарифный план).
- **Пропуски** — есть ли пустые ячейки (здесь их нет).
- **Распределение целевой переменной** — сколько клиентов ушло / осталось.

> **Зачем?** Прежде чем строить модель, нужно понять, с чем мы работаем. Если пропусков много — нужно их заполнить. Если классы сильно несбалансированы — модель может «хитрить» и всегда предсказывать большинство.

#### 2. Подготовка данных (предобработка)
```python
preprocessor = ColumnTransformer(transformers=[
    ('num', StandardScaler(), numeric_features),
    ('cat', OneHotEncoder(...), categorical_features)
])
```
- **StandardScaler** — приводит числовые признаки к одному масштабу (среднее = 0, разброс = 1). Без этого модели вроде KNN будут считать, что «доход в тысячах» важнее «количества звонков в единицах» просто из-за разницы в масштабе.
- **OneHotEncoder** — превращает категориальные признаки (штат, тариф "да/нет") в числа. Например, столбец «тариф» с двумя значениями превращается в столбцы «тариф_да» (1 или 0) и «тариф_нет» (1 или 0).  
  Параметр `drop='first'` убирает один из столбцов, т.к. он избыточен (если «да» = 0, значит «нет» = 1).

> **Зачем ColumnTransformer?** Он позволяет применить разную обработку к разным типам столбцов в одном шаге.

#### 3. Разделение на обучающую и тестовую выборки
```python
X_train, X_test, y_train, y_test = train_test_split(X, y, test_size=0.25, random_state=42)
```
- **75%** данных уходит на обучение модели.
- **25%** — на проверку (тест). Мы «прячем» эти данные от модели, чтобы потом честно оценить, насколько хорошо она предсказывает на новых данных.
- `random_state=42` — фиксируем генератор случайных чисел, чтобы при повторном запуске результат был одинаковым.

#### 4. Обучение трёх моделей
```python
models = {
    'Логистическая регрессия': LogisticRegression(...),
    'Дерево решений': DecisionTreeClassifier(...),
    'KNN': KNeighborsClassifier()
}
```

| Модель | Суть простыми словами |
|---|---|
| **Логистическая регрессия** | Проводит «линию» (или гиперплоскость) между классами. Хорошо работает, когда зависимость линейная. |
| **Дерево решений** | Строит дерево «если-то» вопросов: «Доход > 50?» → «Да» → «Звонил > 10 раз?» → ... Интуитивно понятна. |
| **KNN (K ближайших соседей)** | Смотрит на K самых похожих клиентов из обучающей выборки и «голосует» — к какому классу принадлежит новый. |

Каждая модель **оборачивается в Pipeline** — цепочку «предобработка → модель», чтобы всё работало автоматически в правильном порядке.

#### 5. Оценка и визуализация
- **Accuracy (точность)** — доля правильных ответов. Для каждой модели считается `accuracy_score`.
- **Столбчатая диаграмма** — наглядно сравниваем точность моделей.
- **Confusion Matrix (матрица ошибок)** — для лучшей модели:
  - **True Negatives** — правильно определили лояльных.
  - **True Positives** — правильно определили ушедших.
  - **False Positives** — ошибочно предсказали уход (клиент остался).
  - **False Negatives** — **самая опасная ошибка** — не заметили, что клиент уйдёт.

> **Почему это важно?** Если мы не замечаем уход клиента (False Negative), компания теряет деньги. Именно поэтому в следующих практиках мы будем бороться за повышение метрики **Recall** (полноты).

## Практика 2 — Настройка и улучшение модели

**Файл:** `2/main.py`  
**Задача:** Улучшить модель из Практики 1 — сделать так, чтобы она лучше находила уходящих клиентов.  
**Датасет:** тот же `telecom_churn.csv`

### Проблема, которую мы решаем

В Практике 1 модель хорошо предсказывала лояльных (класс 0), но **плохо находила ушедших** (класс 1). Это связано с **дисбалансом классов** — ушедших клиентов в датасете значительно меньше, чем лояльных. Модель «ленится» и чаще предсказывает большинство.

### Что делает код (пошагово)

#### 1. Диагностика старой модели
```python
old_model = ImbPipeline(steps=[
    ('preprocessor', preprocessor),
    ('classifier', DecisionTreeClassifier(random_state=42))
])
```
Сначала обучаем **ту же модель** (дерево решений без настроек) и печатаем полный `classification_report`:
- **Precision** — из всех, кого модель назвала «ушедшими», сколько реально ушли?
- **Recall** — из всех реально ушедших, скольких модель нашла?
- **F1-Score** — среднее гармоническое Precision и Recall — одно число, которое балансирует оба показателя.

#### 2. SMOTE + GridSearchCV — два ключевых приёма

**SMOTE (Synthetic Minority Over-sampling Technique)**
```python
('smote', SMOTE(random_state=42))
```
- Проблема: ушедших клиентов мало → модель их плохо учит.
- Решение: SMOTE **генерирует синтетические** (искусственные) примеры ушедших клиентов, чтобы выровнять классы.
- Как это работает: SMOTE берёт реального ушедшего клиента, находит его «соседей» среди других ушедших, и создаёт новую точку **между** ними в пространстве признаков.

> **Почему не просто копируем?** Копирование (oversampling) создаёт дубли, и модель может их «заучить». SMOTE создаёт новые, но **похожие** примеры — это лучше для обобщающей способности модели.

**GridSearchCV (Поиск по сетке гиперпараметров)**
```python
param_grid = {
    'classifier__max_depth': [5, 10, 15, 20, None],
    'classifier__min_samples_split': [2, 10, 20],
    'classifier__min_samples_leaf': [1, 5, 10],
    'classifier__criterion': ['gini', 'entropy']
}
```
- У дерева решений есть **гиперпараметры** — настройки, которые мы задаём до обучения:
  - `max_depth` — максимальная глубина дерева (чем глубже — тем сложнее, но риск переобучения).
  - `min_samples_split` — минимальное число примеров для разделения узла.
  - `min_samples_leaf` — минимум примеров в листе.
  - `criterion` — критерий разделения (gini или entropy).
- GridSearchCV **перебирает все комбинации** этих параметров (5 × 3 × 3 × 2 = 90 вариантов) и для каждой проводит **5-Fold кросс-валидацию** (делит обучающую выборку на 5 частей, учит на 4, проверяет на 1-й, повторяет 5 раз).
- Цель — найти комбинацию с лучшим **F1-Score** (баланс Precision и Recall).

#### 3. Финальная оценка
- Строим **Confusion Matrix** для новой модели.
- Строим **ROC-кривую** — график, показывающий, как модель «разделяет» классы при разных порогах. **AUC** (площадь под кривой) — чем ближе к 1, тем лучше. 0.5 — уровень случайного угадывания.
- Выводим **Feature Importance** — какие признаки были самыми полезными для принятия решения (например, количество звонков в сервис).

#### 4. Таблица сравнения
Сравниваем старую и новую модель по всем метрикам. Цель — **повысить Recall** (находить больше ушедших) при сохранении приемлемого качества.

## Практика 3 — Мультиклассовая классификация

**Файл:** `3/main.py`  
**Задача:** Перейти от бинарной классификации (2 класса: ушёл/не ушёл) к **мультиклассовой** (3 и более классов).  
**Датасет:** `Wine` — встроенный датасет scikit-learn с характеристиками итальянских вин трёх сортов.

### Чем отличается от предыдущих работ?

В Практиках 1-2 мы предсказывали **два класса** (ушёл или нет). Здесь — **три класса** (три сорта вина). Это требует:
- Метрик, которые работают с несколькими классами (**Macro F1**).
- Стратегий для балансировки более чем двух классов.

### Что делает код (пошагово)

#### 1. Датасет Wine
```python
wine = datasets.load_wine()
```
- 178 образцов вин, 13 химических признаков (алкоголь, кислотность, зольность...).
- 3 класса: `class_0`, `class_1`, `class_2` — три разных сорта.
- Датасет относительно лёгкий, но хорошо подходит для изучения мультиклассовых подходов.

#### 2. Разведочный анализ (EDA)
- Считаем, сколько образцов в каждом классе → строим **столбчатую диаграмму**.
- Видим, что классы не идеально сбалансированы (59, 71, 48) — но дисбаланс не критичный.

#### 3. Обучение двух моделей
| Модель | Суть |
|---|---|
| **RandomForest (Base)** | Ансамбль из множества деревьев решений. Каждое дерево обучается на случайной подвыборке данных, итоговый ответ — голосованием большинства. Устойчивее одного дерева. |
| **LogisticRegression** | Линейная модель, которая для мультикласса использует стратегию «один против всех» (one-vs-rest). |

- Используем `stratify=y` при разделении — это гарантирует, что в обучающей и тестовой выборках процент каждого класса одинаковый.

#### 4. Балансировка весов
```python
rf_balanced = RandomForestClassifier(random_state=42, class_weight='balanced')
```
Параметр `class_weight='balanced'` автоматически увеличивает «штраф» за ошибки на редких классах. Модель начинает уделять больше внимания малочисленному `class_2`.

> **Отличие от SMOTE:** SMOTE создаёт новые данные, а `class_weight` меняет функцию потерь — модель получает больший «штраф» за ошибку на редком классе. Оба метода борются с дисбалансом, но разными способами.

#### 5. Метрика — Macro F1-Score
```python
f1_score(y_test, y_pred, average='macro')
```
- **Macro F1** — считает F1 для каждого класса отдельно, потом берёт **среднее**.
- Это справедливая метрика для мультикласса: она не позволяет модели «прятаться» за большой класс.

## Практика 4 — Оптимизация мультиклассовой модели

**Файл:** `4/main.py`  
**Задача:** Улучшить модель из Практики 3 с помощью продвинутых методов — SMOTE, кривых обучения, GridSearchCV.  
**Датасет:** тот же `Wine`

### Что делает код (пошагово)

#### 1. SMOTE для мультикласса
```python
smote = SMOTE(random_state=42)
X_train_smote, y_train_smote = smote.fit_resample(X_train, y_train)
```
- SMOTE работает и для 3+ классов: он доводит каждый класс до размера самого большого.
- Печатаем распределение **до** и **после** — видим, что теперь все классы имеют одинаковое количество примеров.

#### 2. Кривые обучения (Learning Curve) — диагностика модели
```python
train_sizes, train_scores, test_scores = learning_curve(
    RandomForestClassifier(...), X_train_smote, y_train_smote,
    cv=5, scoring='f1_macro', train_sizes=np.linspace(0.1, 1.0, 5)
)
```
- Строим график: как меняется качество модели при увеличении объёма обучающих данных.
- **Обучающая кривая** (синяя) — качество на тех данных, на которых учились.
- **Валидационная кривая** (оранжевая) — качество на данных, которые модель не видела.

**Как читать график:**
- Если **обе кривые высоко и рядом** → модель хорошая, данных достаточно.
- Если **обучающая высоко, валидационная низко** → **переобучение** (модель заучила тренировочные данные, но не умеет обобщать).
- Если **обе низко** → **недообучение** (модель слишком простая).

#### 3. Pipeline + GridSearchCV
```python
pipeline = ImbPipeline(steps=[
    ('scaler', StandardScaler()),
    ('rf', RandomForestClassifier(random_state=42))
])
param_grid = {
    'rf__max_depth': [5, 10, None],
    'rf__n_estimators': [50, 100]
}
```
- `StandardScaler` — нормализация признаков (важно для стабильности).
- `n_estimators` — количество деревьев в лесу (больше = точнее, но медленнее).
- `StratifiedKFold` — кросс-валидация, которая **сохраняет пропорции классов** в каждой «складке».

#### 4. Итоговая таблица
Сравниваем Macro-F1 на трёх этапах:
1. Базовая модель (из практики 3).
2. + SMOTE (без тюнинга).

## Практика 5 — Регрессия

**Файл:** `5/main.py`  
**Задача:** Перейти от **классификации** (предсказание категории) к **регрессии** (предсказание числа). Прогнозируем стоимость домов.  
**Датасет:** `California Housing` — встроенный датасет scikit-learn с данными о недвижимости в Калифорнии.

### Чем регрессия отличается от классификации?

| | Классификация | Регрессия |
|---|---|---|
| **Что предсказываем** | Категорию (ушёл/не ушёл, сорт вина) | Число (цена дома) |
| **Метрики** | Accuracy, F1, Recall | MAE, MSE, R² |
| **Пример** | «Этот клиент уйдёт» | «Этот дом стоит $350k» |

### Что делает код (пошагово)

#### 1. Исследование целевой переменной
```python
sns.histplot(y, bins=50, kde=True)
```
- Строим **гистограмму** — распределение цен на дома.
- Видим, есть ли «выбросы» (слишком дорогие дома) и насколько данные «нормальные».

#### 2. Выбор признаков
```python
selected_features = ['MedInc', 'AveRooms']
```
- **MedInc** — медианный доход в районе.
- **AveRooms** — среднее количество комнат.
- Берём только два признака для наглядности (легко визуализировать на 2D-графике).
- Строим **scatter-plot** — как каждый признак связан с ценой.

#### 3. Линейная регрессия
```python
lr = LinearRegression()
```
- Самая простая модель регрессии: ищет **прямую линию** (в многомерном пространстве — гиперплоскость), которая лучше всего приближает данные.
- Формула: `Цена = a₁×Доход + a₂×Комнаты + b`, где `a₁`, `a₂` — коэффициенты, `b` — смещение (intercept).

**Метрики регрессии:**
| Метрика | Что означает |
|---|---|
| **MAE** (Mean Absolute Error) | Средняя абсолютная ошибка — «в среднем мы ошибаемся на X тысяч долларов» |
| **MSE** (Mean Squared Error) | Средняя квадратичная ошибка — штрафует большие ошибки сильнее |
| **R² (коэффициент детерминации)** | Доля дисперсии, объяснённая моделью. 1.0 = идеально, 0 = модель не лучше среднего |

**Интерпретация коэффициентов:**
```
Коэффициент [MedInc]: 0.84   → рост дохода на 1 единицу → цена растёт на $84k
Коэффициент [AveRooms]: -0.04 → доп. комната → цена падает на $4k (парадокс — скорее всего, корреляция с другими факторами)
```

#### 4. Сравнение с деревом решений
```python
dt = DecisionTreeRegressor(max_depth=3, random_state=42)
```
- Дерево решений работает и для регрессии — вместо «голосования» по классам оно усредняет значения в каждом листе.
- `max_depth=3` — ограничиваем глубину, чтобы не переобучалось.

#### 5. Scatter-plot: реальная цена vs предсказанная
- Идеальная модель → все точки лежат на диагонали `y = x`.
- Чем дальше точки от диагонали — тем больше ошибка.

## Практика 6 — Борьба с переобучением

**Файл:** `6/main.py`  
**Задача:** Показать, что такое **переобучение**, и как его побороть с помощью **Ridge-регуляризации**.  
**Датасет:** тот же `California Housing`

### Что такое переобучение?

Модель **слишком хорошо выучила обучающие данные** (включая шум и случайности), но плохо работает на новых. Аналогия: студент, который заучил ответы прошлогодних билетов, но не понимает предмет.

### Что делает код (пошагово)

#### 1. Базовая модель (ориентир)
```python
lr_base = LinearRegression()
lr_base.fit(X_train_scaled, y_train)
r2_base_test = r2_score(y_test, lr_base.predict(X_test_scaled))
```
- Обучаем модель на всех 8 реальных признаках, запоминаем R² на тесте как **базовый уровень**.

#### 2. Искусственная «поломка» — добавляем шум
```python
noise_train = np.random.normal(0, 1, (X_train.shape[0], 30))
X_train_noisy = np.hstack((X_train, noise_train))
```
- Генерируем **30 случайных (бессмысленных) столбцов** и добавляем к данным.
- Теперь вместо 8 признаков → 38, но 30 из них — чистый шум.

> **Зачем?** Это моделирует реальную ситуацию, когда в данных есть признаки, которые **не связаны с целевой переменной**, но модель пытается использовать и их.

#### 3. Демонстрация переобучения
```python
lr_broken = LinearRegression()
lr_broken.fit(X_train_noisy_scaled, y_train)
```
- R² на **обучающей** выборке — высокий (модель «подстроилась» под тренировочные данные, включая шум).
- R² на **тестовой** выборке — резко падает! Модель запомнила случайный шум и не умеет обобщать.

#### 4. Спасение — Ridge-регрессия
```python
ridge = Ridge(alpha=a, random_state=42)
```

**Ridge-регрессия** — это линейная регрессия с **регуляризацией**:
- Обычная линейная регрессия минимизирует ошибку: `MSE`.
- Ridge минимизирует: `MSE + α × (сумма квадратов коэффициентов)`.
- Параметр **α (alpha)** — «сила» регуляризации:
  - α = 0 → обычная линейная регрессия.
  - α → ∞ → все коэффициенты стремятся к нулю (модель ничего не предсказывает).
  - Оптимальное α — **компромисс** между точностью и простотой.

**Как это помогает?**
Ridge **штрафует большие коэффициенты**. Шумовые признаки получают маленькие коэффициенты → модель их фактически игнорирует → переобучение уменьшается.

#### 5. Перебор alpha
```python
alphas = [0.1, 1, 10, 100, 1000]
for a in alphas:
    ridge = Ridge(alpha=a)
    ...
```
Перебираем разные значения alpha и находим оптимальное по R² на тестовой выборке.

#### 6. Финальная таблица
| Модель | R² на тесте | Комментарий |
|---|---|---|
| Базовая (без шума) | ~0.61 | Ориентир |
| Сломанная (с шумом, без Ridge) | ~0.57 | Переобучение! |
| Исправленная (Ridge) | ~0.61 | Спасение! |

```
Практика 1: Знакомство → выбираем лучшую из трёх простых моделей
       ↓
Практика 2: Улучшаем → SMOTE (данные), GridSearch (параметры)

       ↓

       ↓

```

## Словарь терминов

| Термин | Простое объяснение |
|---|---|
| **Accuracy** | Доля правильных ответов из всех |
| **Precision** | «Из тех, кого назвали ушедшими, сколько реально ушли?» |
| **Recall** | «Из всех реально ушедших, скольких нашли?» |
| **F1-Score** | Баланс между Precision и Recall |
| **Confusion Matrix** | Таблица: что модель предсказала vs что было на самом деле |
| **ROC-AUC** | Площадь под кривой разделения классов (0.5 = случайно, 1.0 = идеально) |
| **MAE / MSE** | Средние ошибки для регрессии (абсолютная / квадратичная) |
| **R²** | Насколько модель объясняет данные (0 = плохо, 1 = отлично) |
| **SMOTE** | Генерация синтетических примеров для балансировки классов |
| **GridSearchCV** | Автоматический перебор параметров модели с кросс-валидацией |
| **Кросс-валидация** | Проверка модели на нескольких разбиениях данных |
| **Pipeline** | Цепочка шагов обработки данных + обучения в одном объекте |
| **Переобучение** | Модель запомнила данные вместо того, чтобы научиться обобщать |
| **Регуляризация** | Штраф за сложность модели, помогает против переобучения |
| **StandardScaler** | Нормализация данных к среднему 0 и разбросу 1 |
| **OneHotEncoder** | Превращение категорий в числовые столбцы |
